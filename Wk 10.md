# Week 10 Revision Notes – Software Deployment & Monitoring

## 1. Software Deployment

### Definition

**Deployment** is the process of making a software application available for users by installing and configuring it in an execution environment.

### Monitoring

**Monitoring** is the process of collecting and analyzing information about a running application to ensure it performs correctly and efficiently.

Deployment answers:

> **Where does the software run?**

Monitoring answers:

> **How well is the software running?**



---

# 2. Deployment Environments

Software typically passes through four environments.

## A. Development Environment

### Purpose

Where developers write and test code.

Contains

* IDE
* Local database
* Working copy of code
* Build tools

Used by developers only.

---

## B. Testing Environment

Purpose

Verify correctness after integrating code.

Characteristics

* Used by testers
* Bugs are identified
* Not accessible to customers

---

## C. Staging Environment

### Definition

A staging environment is a **production-like environment** used to verify deployment before releasing to real users.

Characteristics

* Closely resembles production
* Runs on remote servers
* Final validation before release

Why needed?

* Test deployment scripts
* Verify configurations
* Preview new features
* Performance testing
* Prevent production failures

---

## D. Production Environment

The live environment used by customers.

Characteristics

* Highest reliability required
* Real users
* Real data
* Downtime should be minimal



---

# Development vs Testing vs Staging vs Production

| Environment | Used By         | Purpose                  |
| ----------- | --------------- | ------------------------ |
| Development | Developers      | Coding                   |
| Testing     | Testers         | Bug detection            |
| Staging     | Developers & QA | Final deployment testing |
| Production  | Customers       | Actual use               |

---

# 3. Deployment Strategies

## A. Blue-Green Deployment

### Idea

Maintain **two production environments**.

* Blue = Current version
* Green = New version

Traffic is switched from Blue to Green when ready.

### Advantages

* Easy rollback
* Zero/minimal downtime
* Safe deployment

### Disadvantage

Requires duplicate infrastructure.

---

## B. Canary Deployment

### Idea

Release the new version to **only a small percentage of users**.

Users may be selected

* randomly
* by location
* by demographics
* by user profile

If successful,

gradually increase rollout.

### Advantage

Limits impact of bugs.

### Disadvantage

Need to manage multiple versions simultaneously.

---

## C. Versioned Deployment

### Idea

Keep multiple versions running simultaneously.

Users choose which version to use.

### Advantage

Users can migrate gradually.

### Disadvantage

Must maintain multiple versions.



---

# Deployment Strategy Comparison

| Strategy   | Key Idea                        | Major Advantage           |
| ---------- | ------------------------------- | ------------------------- |
| Blue-Green | Two production environments     | Easy rollback             |
| Canary     | Small percentage of users first | Reduced deployment risk   |
| Versioned  | Multiple versions coexist       | User-controlled migration |

# 4. Hosting Options

## A. Bare Metal Servers

Own physical servers.

Advantages

* Highest performance
* Complete control

Disadvantages

* Expensive
* Maintenance responsibility

---

## B. Infrastructure as a Service (IaaS)

Provider supplies virtual machines.

Examples

* AWS EC2
* DigitalOcean
* Linode

Advantages

* Lower cost
* No hardware maintenance

Disadvantages

* Configuration responsibility remains
* Shared resources

---

## C. Platform as a Service (PaaS)

Provider manages operating system and platform.

Examples

* Heroku
* Google App Engine

Advantages

* Very easy deployment
* Little administration

Disadvantages

* Less control
* Platform restrictions



---

# Bare Metal vs IaaS vs PaaS

| Bare Metal          | IaaS               | PaaS                      |
| ------------------- | ------------------ | ------------------------- |
| Own hardware        | Virtual machines   | Complete platform         |
| Highest control     | Medium control     | Least control             |
| Highest maintenance | Medium maintenance | Minimal maintenance       |
| Most expensive      | Moderate           | Usually cheapest to start |

# 5. Continuous Integration (CI)

### Definition

Continuous Integration (CI) is the practice of **frequently integrating code changes into a shared repository and automatically building and testing them.**

Typical CI workflow

1. Developer commits code
2. CI server detects changes
3. Build starts automatically
4. Automated tests run
5. Deploy to staging
6. Ready for production



---

# Best Practices for Continuous Integration

1. Maintain a single source repository.
2. Automate the build.
3. Make builds self-testing.
4. Commit to the main branch every day.
5. Build every commit on the CI server.
6. Fix broken builds immediately.
7. Automate deployment.



---

# Continuous Integration Server

Purpose

Automatically

* monitors repository
* builds software
* executes tests
* reports failures

Examples

* Jenkins
* CruiseControl



---

# Benefits of Continuous Integration

### For Developers

* Early bug detection
* Smaller merge conflicts
* Modular development
* Faster feedback

### For Organization

* Faster deployment
* Less release-day stress
* Stable software

### For Customers

* Faster feature availability
* Early prototypes

Drawback

* Initial setup effort

# 6. Improving Performance

## A. Caching

### Definition

Store frequently used results in memory.

Instead of

Database

↓

Use

Memory (Cache)

Examples

* Redis
* Memcached

Levels

* Browser cache
* Web server cache
* Database query cache

Benefit

Much faster response.



---

## B. Task Queues

### Definition

Move long-running tasks outside the request-response cycle.

Examples

* Sending emails
* Notifications
* Bulk database updates

Advantages

* Faster web response
* Better scalability



---

## C. Faster Page Loading

Techniques

### Minification

Remove unnecessary

* spaces
* comments
* formatting

Result

Smaller files.

---

### Compression

Compress static content before sending.

Examples

* Gzip
* Brotli

Benefits

* Reduced bandwidth
* Faster downloads

# 8. Customer Behaviour Analytics

Applications monitor user behaviour using metrics such as:

### Clickstream

Sequence of pages visited.

---

### Dwell Time

Time spent on a page.

---

### Abandonment

Percentage of users leaving before completing a process.

Example

Product

↓

Address

↓

Payment

↓

Confirmation

Users leaving before payment indicate abandonment.

Analytics help improve the user experience.

# High-Probability Comparisons

### Staging vs Production

| Staging                | Production                 |
| ---------------------- | -------------------------- |
| Testing before release | Live system                |
| Internal users         | Real customers             |
| Safe experimentation   | High availability required |

---

### Blue-Green vs Canary

| Blue-Green                     | Canary                   |
| ------------------------------ | ------------------------ |
| Entire traffic switches        | Partial traffic first    |
| Fast rollback                  | Lower deployment risk    |
| Needs duplicate infrastructure | Needs traffic management |

---

### IaaS vs PaaS

| IaaS                      | PaaS                    |
| ------------------------- | ----------------------- |
| Manage OS and application | Manage only application |
| More control              | Easier deployment       |
| More configuration        | Less flexibility        |

---

### Caching vs Task Queue

| Caching               | Task Queue                   |
| --------------------- | ---------------------------- |
| Speeds repeated reads | Handles long-running work    |
| Stores data in memory | Executes jobs asynchronously |
